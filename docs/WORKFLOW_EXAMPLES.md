# ワークフロー例

1日、1週間、プロジェクト全体を通じてPKMシステムを効果的に使用する方法を示す実世界のシナリオ。

## 日次ワークフロー

### 朝のルーティン (15分)

#### 6:00 AM - 起床
```bash
# まず、コンテキストを読み込む
claude /onboard

# 今日のデイリーノートを作成
claude /daily
```

#### 6:05 AM - レビューと計画
Claudeが以下を支援します:
1. 昨日の未完了タスクをレビュー
2. 固定の予定のためにカレンダーを確認
3. 一つの大きな優先事項を特定
4. 現実的なタイムブロックを設定

#### 6:10 AM - 意図の設定
```markdown
## 今日のフォーカス
**一つのこと:** プロジェクト提案を完成させる

## 意図
- 会議で集中する
- 90分ごとに休憩を取る
- 18時までに仕事を終える
```

#### 6:15 AM - 開始準備完了
```bash
# オプション: モチベーションを得る
claude "現在の目標に関連するモチベーション引用をください"
```

### 正午のチェックイン (5分)

#### 12:00 PM - 進捗レビュー
```markdown
## 正午チェック
- ✅ 完了: 4/8タスク
- 🔄 進行中: プロジェクト提案 (70%)
- ⚡ エネルギーレベル: 7/10
- 🎯 午後のフォーカス: クライアント会議準備
```

### 夜のシャットダウン (10分)

#### 5:30 PM - 片付け
1. 完了したタスクをチェック
2. 未完了項目とその理由を記録
3. 散らばった思考をキャプチャ

#### 5:35 PM - 振り返り
```markdown
## 1日の終わりの振り返り
### うまくいったこと
- 提案を予定より早く完了
- デザイン問題で素晴らしいブレークスルー

### 改善できること
- コンテキストスイッチが多すぎた
- 昼休みをスキップした

### 明日の最優先事項
- チームと提案をレビュー
```

#### 5:40 PM - 保存と同期
```bash
claude /push "日次作業完了 - {{date}}"
```

## 週次ワークフロー

### 日曜日の週次レビュー (30分)

#### ステップ1: レビューの作成 (5分)
```bash
claude /weekly
```

#### ステップ2: 先週のレビュー (10分)
各デイリーノートを確認:
```bash
claude "先週のデイリーノートから達成事項を要約して"
```

パターンを探す:
- 最も生産的だった時間帯は?
- 計画を狂わせたものは?
- 祝福できる勝利は?

#### ステップ3: 来週の計画 (10分)
```markdown
## 来週の計画
### 一つの大きなこと
他に何もしなくても、これをする: [Q1レポートを完成]

### 主要プロジェクト
1. **プロジェクトアルファ** - マイルストーン: デザインレビュー
2. **スペイン語学習** - ゴール: 第4章完了
3. **健康** - 目標: 4回のワークアウト

### カレンダーブロック
- 月 9-11 AM: レポートの深い作業
- 水 2-4 PM: チーム計画
- 金 3-5 PM: 週次レビューと計画
```

#### ステップ4: クリーンアップ (5分)
```bash
# 古いノートをアーカイブ
claude "30日より古いデイリーノートをArchivesに移動して"

# プロジェクトステータスを更新
claude "すべてのアクティブなプロジェクトをレビューしてステータスを更新して"

# すべてをコミット
claude /push "週次レビュー完了 - 第{{week_number}}週"
```

### 月次レビュー (1時間)

#### 月の第一日曜日
```bash
# 完全なコンテキストを読み込む
claude /onboard all

# 月次レビューを作成
claude "年間目標への進捗を分析する月次レビューを作成して"
```

レビュープロセス:
1. **定量分析** (20分)
   - 完了したタスクをカウント
   - 目標進捗を測定
   - 習慣の一貫性を追跡

2. **定性的振り返り** (20分)
   - どんなテーマが現れた?
   - 何に驚いた?
   - 注意が必要なパターンは?

3. **計画の調整** (20分)
   - 月次目標を更新
   - プロジェクトの優先順位を修正
   - 日次ルーティンを調整

## プロジェクトワークフロー

### 新しいプロジェクトの開始

#### ステップ1: 構造の作成
```bash
claude "標準構造で 'Website Redesign' という新しいプロジェクトを作成して"
```

#### ステップ2: 成功の定義
```markdown
# プロジェクト: ウェブサイトリデザイン

## 成功基準
1. 3月1日までに新デザイン公開
2. ページロード時間 < 2秒
3. モバイルファーストのレスポンシブデザイン
4. アクセシビリティスコア > 95

## マイルストーン
- [ ] 第1週: 要件収集完了
- [ ] 第2週: ワイヤーフレーム完成
- [ ] 第3週: デザイン承認
- [ ] 第4週: 開発開始
```

#### ステップ3: プロジェクトCLAUDE.mdの作成
```markdown
# Claudeのためのコンテキスト: ウェブサイトリデザイン

## プロジェクトステータス
現在計画フェーズ

## 主要な決定事項
- React + Next.jsを使用
- Vercelでホスティング
- デザインシステム: Tailwind

## 支援する際
- アクセシビリティを思い出させる
- パフォーマンス最適化を提案
- モバイルファーストを念頭に置く
```

## モバイルワークフロー

### 外出先でのタスクキャプチャ

#### GitHub Issues経由
1. GitHubモバイルアプリを開く
2. タイトルでissueを作成: "Task: [説明]"
3. ラベルを追加: "task"
4. Claudeが自動的にデイリーノートに追加

#### ボイスメモ経由
1. ボイスメモを録音
2. GitHubにissueとして共有
3. Claudeが文字起こしと処理

### モバイル週次レビュー

#### スマートフォンから
1. GitHubモバイルを開く
2. 最近のコミットをレビュー
3. issueを作成: "Weekly review needed"
4. Claudeがレビューテンプレートを生成

### 緊急アクセス

#### デスクトップが利用できない場合
```
1. 任意のブラウザからGitHub.comにアクセス
2. Vaultリポジトリに移動
3. ブラウザで直接ファイルを編集
4. デスクトップに戻ったときに変更が同期
```

## ワークフロー成功のヒント

1. **小さく始める**: 複雑さを追加する前にデイリーノートをマスター
2. **一貫性を保つ**: 同じ時間、同じプロセス、毎日
3. **週次で反復**: うまくいかないものを調整
4. **パターンを追跡**: 何が助けになり、何が妨げになるかに注意
5. **徐々に自動化**: パターンが安定したら自動化を追加

覚えておいてください: ワークフローは摩擦を減らすべきであり、増やすべきではありません。何かが重く感じたら、それを単純化してください。
